# ことば品質チェック（kotoba-content-review）

ことばパズルの**英文と和訳の自然さ**を、毎回**同じ固定観点**で点検・修正する。
機械検査（`kotoba-sentence-add` の band/文法/漢字/重複）は通っても、**「子が言わない英文」「直訳調・誤訳の和訳」は素通り**する。これはアプリの根幹品質に直結するので、**人＋AIの目で固定ルーブリックを必ず通す**。

> 関連：文を増やす手順＝[[kotoba-sentence-add]]（このスキルはその「自然さレビュー」を独立・厳密化したもの）。背景・実例＝メモ [[age-tiered-content-generation]]。

## 大原則
- **機械が拾えない品質を人＋AIで担保**。band/文法/漢字/重複は CLI、**自然さはこのスキル**。
- **二重チェックで漏れを減らす**：①私(Claude)が全件ルーブリック点検 → ②codex に同じ全件を独立監査させる → ③突き合わせて拾い直す。1人の目視は必ず漏れる（実例：初回は4件しか拾えず取りこぼした）。
- **迷ったら声に出して読む**。違和感＝直す or 捨てる。
- **英文の弱さは英文ごと直す**（和訳だけ直しても根本解決しない）。en を変えると id(UUIDv5) が変わるが決定論なのでOK。**変更後は必ず `kotoba-sentence-add` の機械検査(band/文法/漢字/重複)を再通過**させる（自然さを直して壁を破らないこと）。
- 書き出し作業は**ブランチ/ワークツリー**で（`main` 直接禁止）。スキル本体(`.claude/`)編集は main 可。

## ルーブリック（✗→✓・必ず全項目）

### A. 英文の自然さ（生成段階で弾く・和訳以前）
**判定軸＝「日本の子が実際に使う場面が浮かぶか」「ネイティブが言う言い回しか」。** 文法が正しくてもダメなものはダメ。
- ✗ `I am sorry now`（now が浮く・場面不明） → ✓ `I am sorry` / `Sorry about that`
- ✗ `Help your good friend`（命令文＋場面不明） → ✓ `I help my friend` / `Be kind to your friend`
- ✗ `Sora has two long legs`（誰でも2本・描写が不自然） → ✓ `Sora runs very fast`
- ✗ `Thank you, good bird`（good bird が不自然） → ✓ `Thank you, little bird`
- ✗ 飾り文 `This is a pen`（覚える邪魔・つまらない） → ✓ 役立つ会話 or ユーモア（spec準拠）
- チェック：①場面が浮かぶ ②形容詞の乗せ方が自然(good/nice の多用に注意) ③段階の文法天井内 ④こわくない/きたなくない/いじわるでない。

### B. 和訳の自然さ（直訳調をなくす）
1. **直訳禁止**：語順・冠詞・所有格をそのまま訳さない。呼びかけ `my friend`→「わたしの おともだち」のような機械訳をしない。
   - ✗「わたしの おともだち」 → ✓「ねえ」「きみ」省略 等、文脈で自然に
2. **形容詞＋名詞**：`good/nice` を「よい/いい」にそのまま乗せない。
   - ✗「よい おともだち」→✓「なかよしの／だいじな おともだち」／✗「いい ことりさん」→✓「ことりさん」
3. **定型句は定型訳**：
   - `No thank you`→「いいえ けっこうです」(✗「いいえ ありがとう」)／`You're welcome`→「どういたしまして」／`Have a nice day`→「よい いちにちを」／`Excuse me`→「すみません」
4. **表現の統一**：同じ英語表現は全文で同じ訳に（`No thank you` の訳がバラつかない・`Thank you` の訳ゆれを揃える）。
5. **多義語の誤訳**：特に**動物の動作**。
   - ✗「うさぎが とんでいる」(jump を fly と誤訳) → ✓「うさぎが はねている」／ jump=はねる, fly=とぶ, swim=およぐ, run=はしる を取り違えない
6. **指示詞**：`These/Those`→「これら/それら」は子に不自然。
   - ✗「これらは わたしの 目です」→✓「これ、わたしの 目だよ」
7. **語尾の統一**：です・ます と だ・よ を混ぜない。子向けは親しみやすい口調で**1文内・近接で統一**。
8. **子ども口調・ふりがな調**：硬い書き言葉でなく話し言葉。漢字は段階上限内（[[KanjiLevelGate]]・段階aはひらがな）。

## 手順（監査→作り直し）

1. **全文ダンプ**：`scripts/person_templates.authoring.json` から `id / en / ja` を全件出す（python）。
2. **私が全件ルーブリック点検**：A/B を1文ずつ。`status=ok | fix-ja | fix-en | drop` と「直す案(en/ja)」を表に。
3. **codex 二重チェック（必須）**：同じ全件を渡し、`mcp__codex__codex`(read-only) で「日本の子向け英語教材として、英文と和訳それぞれ不自然なものを全部挙げて。✗理由＋✓修正案」。私の表と**突き合わせて拾い直す**（codex が拾い私が見落としたものを必ず反映）。
4. **作り直し**：`fix-en` は英文ごと書き換え（機能/slot/category 維持・段階の壁を守る）。`fix-ja` は和訳のみ。`drop` は素から除く（未公開なので自由）。
   - en を変えたら **`kotoba-sentence-add` 手順A 4〜6（プレビュー→却下0→`--write`→決定論）** を必ず通す。ja のみなら `--write` 再生成で id 不変・ja 行だけ差分。
   - authoring は **`scripts/` と `iPadPrototype/Resources/` の2ファイルを byte一致**で。
5. **検証**：`swift test` 緑＋アプリビルド。差分が素直か（ja のみ/en 変更で id 入替）。
6. **コミット〜マージ〜後片付け**は CLAUDE.md の流れで**1行確認してから**。

## 固定の合否ライン
- **英文**：場面が浮かばない/言わない文＝不合格（fix-en か drop）。
- **和訳**：声に出して違和感＝不合格（fix-ja）。直訳調・定型崩れ・多義語誤訳・これら/それら・語尾混在は機械的に不合格。
- **二重チェックを通っていない同梱は不可**（私だけのレビューで `--write` しない）。
