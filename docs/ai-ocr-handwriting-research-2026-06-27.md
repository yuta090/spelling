# AI手書きOCR導入 — 方針・コスト試算・実証テスト計画

作成: 2026-06-27 / 対象: SpellingTrainer（日本人の子ども向け英単語スペル練習 iPad アプリ）

このドキュメントは、手書き採点を「ローカルOCR」から「AIビジョンモデル（VLM）」に引き上げる検討の記録。
背景となるマーケ判断 → 機能の位置づけ → エスカレーション設計 → モデル料金 → 現実的なコスト試算 → ローカルLLM評価 → 実証テスト計画、の順でまとめる。

---

## 0. ターゲット年齢の方針（決定事項）

- このアプリの上限は **中学受験〜高校受験あたりまで** とする。
- 高校生は **同一プロダクトでは狙わない**（ペルソナ・難易度・ブランドの3つ全部がズレるため）。狙うならコアエンジン（`SpellingSyncCore`）を再利用した **別ブランド（受験/英検スペル特化）** として切り出す。
- 手書きOCR機能の価値は **低年齢ほど高い**（運筆・字形がそもそも学習目標で、字も最も汚い＝ローカルOCRが最も死ぬ場所）。中学受験までが本丸、高校受験はおまけ上限の重み付け。

## 1. AI手書きOCRの位置づけ

### 解く課題
- **親（manager）の採点負担をゼロにする** = 忙しい親に刺さる中心価値。
- ローカルOCR（Apple Vision / VNRecognizeText）は **低学年の汚い手書きを認識しきれない** のが実機の実感。これがAI導入の動機。

### コアになるリフレーム（重要）
> AIでも認識できないものは「きれいに書き直そう」と促す

- 技術的弱点（認識失敗）を **教育価値に反転** させる設計。
- 「AIが読めない字＝先生も読めない字＝テストでも×になる字」なので、**読めない→直すべき、は教育的に正しい**。
- これにより **100%の認識精度を追わなくてよくなる**（開発もコストもラクになる）。認識失敗は「失敗」ではなく「次の練習課題」。

## 2. エスカレーション設計（精度とコストを同時に解く）

「全部AIに投げる」とコストが青天井になるため、段階的に上げる。

```
① ローカルVision OCR（無料・即時）
     ├─ 自信高 & 綴り一致 → ◯ そのまま採点（タダ）
     └─ 自信低 or 不一致
            ↓
② クラウドAI Vision（Gemini/GPT/Claude）で再判定（ここだけ課金）
     ├─ 読めた → 採点 ＋「おしい！ここが違うよ」
     └─ 読めない
            ↓
③ 「きれいに書き直してみよう」コーチング（タダ・教育価値◎）
```

- 綺麗な字はローカルで無料処理、**API課金は"判定が割れた難しい字"だけに圧縮**。
- 子どもの大半の回答は正解 or 明確なミスなので、②に落ちる割合（= API送り率）は **2〜3割** に抑えられる想定。

### 純粋ロジックの置き場所
- 判定ロジック（局所OCR結果の確信度判定、綴り一致、エスカレーション可否、確信度に基づく採点）は **`SpellingSyncCore`（SwiftPM）に純粋関数として置く** → `swift test` で回す。
- アプリ本体（画像入力・API I/O・UI）は薄く保つ。

## 3. プライバシーの扱い（地雷除去であって訴求ではない）

- 自社DB保存と第三者AI APIへの送信は、親の心理でも法務（App Store審査）でも別物として扱われる。特に「子どものデータ」は説明義務が出る。
- ただし **普通の親はほぼ気にしない**。プライバシーは"攻めの訴求"ではなく **"地雷除去"** の温度感で扱う。
- 送るなら: 親同意・画像の最小化（タイトにクロップ／特徴量のみ送る等）・保存しない、を明記。

## 5. コスト試算（現実的な利用モデル）

### 1回の「判定」のトークン構成
- 単語1つの手書き画像（小さくクロップ）＋プロンプト＋短い出力。
- 画像トークン: 単語クロップ（例 600×200px）で Claude 約160 / Gemini・GPT 約260。プロンプト ~100、出力 ~20。
- → 入力 約300〜400 / 出力 約20 トークン。

### 1判定あたりの単価
| モデル | 1判定 | （円・¥150/$換算） |
|---|---|---|
| Gemini Flash-Lite | 約 $0.00004 | 約 0.006円 |
| GPT-5.4 nano | 約 $0.0001 | 約 0.016円 |
| Claude Haiku 4.5 | 約 $0.0004 | 約 0.06円 |

### 現実的な利用量（子1人・1学習日）
| 行動 | 判定数 |
|---|---|
| 新規練習 20語 × 3回 | 60 |

| テスト 20語 × 1回 | 20 |
| **計** | **約 100 判定/日** |

月20日アクティブ → **約 2,000 判定/月/人**。

### 月コスト/人
| | エスカレーション設計<br>(API送り25% = 500回) | 全件API送り<br>(2,000回) |
|---|---|---|
| Gemini Flash-Lite | **約3円** | 約12円 |
| GPT-5.4 nano | **約7.5円** | 約30円 |
| Claude Haiku 4.5 | **約30円** | **約120円** |

### 結論
- **保護者プラン月数百円に対して、API原価は基本的に誤差。** 回数制限を表に出さずサブスク内包で問題ない。
  - 子は「今日はもうAI採点できません」が一番萎えるので、**回数制限は見せない設計**にする。
- 例外: **Haiku × 全件送り（約120円/月）** だけは薄利だと無視できない水準。
  → **エスカレーション設計が料金面でも効くのはHaiku採用時**。nano/Flash-Lite なら設計に関係なく誤差。
- 前提（タイトな単語クロップ）が崩れて**ページ全体や大きい画像を送ると画像トークンが増える**ので、クロップは必ずタイトに。

## 6. ローカルLLM（端末内VLM）の評価

- **素の Apple Vision OCR（VNRecognizeText）**: 無料・即時だが、**低学年の汚い手書きに弱い**（活字・整った字向け）。実機の実感どおり。
- **端末内VLM（Qwen3-VL 4B/8B 等）**: 手書き精度は Apple Vision より上がる（Qwen3-VL-8B は OmniDocBench 94.5%）。ただし代償が重い:
  - モデル4〜8GBでアプリ肥大、iPadのRAM/バッテリーを食う、推論が遅い（**即フィードバック原則に反する**）。
  - 継続的な最適化・メンテ工数。
- **判断: クラウドVLMが月数円で済む以上、ローカルVLMは"オフライン必須 or 画像を一切外に出さない"が固い要件のときだけ。** 今回は「普通の親はプライバシーを気にしない」ので **クラウドVLM一択**。ローカルは将来のオプション。

## 7. 実証テスト計画

### 目的
候補モデル（GPT-5 nano / Gemini Flash-Lite / Claude Haiku 4.5、必要なら Sonnet）を、**同じ画像セットで横並び比較**し、精度・誤り種別・レイテンシで意思決定する。

### 方法論（ここを固めるのが肝）
1. **同一画像に全モデルを通す**。子に都度書かせて各モデルで別々に試すと字が毎回違って比較にならない。**1枚撮る→全候補に同じ画像を投げる**。
2. **正解ラベル付きの固定テストセット**（画像＋その単語）を作る。まず30〜50枚。
3. **サンプルの多様性**: 自分の子だけ（n=1）だと一番効かせたい低学年の汚い字を外す risk。可能なら **複数人・年齢バラけ**、Pencil入力と紙＋カメラの両方。他人の子は親の同意を取る。
4. **わざと判読不能サンプルを混ぜる**（グチャグチャの字）。

### 測る指標
| 指標 | 意味 | 重要度 |
|---|---|---|
| **誤受理 (False Accept)** | スペルミスを「正解」にする | **最重要**（親の信頼が即死） |
| 誤拒否 (False Reject) | 正しいのに「違う」 | 高（子のストレス） |
| 判読不能の正答 | 読めない字を捏造せず「読めない」と返せるか | 高（「書き直そう」導線の前提） |
| **コメント品質** | 「おしい！ここが違うよ」等の一言を**安いモデル（nano/Flash-Lite）が出せるか**。Haiku固定にしない | 高（モデル選定を分ける根拠になる） |
| レイテンシ | 即フィードバックに耐えるか | 中 |
| 1判定コスト | 実測トークンでの原価 | 低（誤差） |

> **コメント生成は独立の検証項目**。OCR採点とコメント生成を同じプロンプトで一括させ、まず nano/Flash-Lite で品質が実用域か見る。落ちる場合のみ Haiku 等にフォールバック、あるいはコメントだけ別モデルに分離する設計を検討。

### 進め方
- 各モデルAPIに同一画像を投げて結果をCSVで集計 → 指標を数値で比較。実装は `scripts/ocr-bench/`（OpenRouter経由・bench.py）。
- 「子に書かせて試す」は◎だが、**ラベル付き固定画像セットで全モデル横並び**にして初めて意思決定に使えるデータになる。

### 収集とラベリングの設計（実使用で集める／手入力をほぼ消す）
アプリで普通に使ってもらったものを試験したい。ただし**実使用の画像には「実際に何と書いたか(ground_truth)」が付かない**ので、そこを設計で解く。

#### 最良案：リモート採点（親採点）データに相乗り 〔2026-06-27 追記〕
関連: [remote-grading-spec.md](./remote-grading-spec.md)。「親が別端末で子の書き取りを採点する」リモート採点機能が、ベンチに必要なものをほぼ全部すでに持っている。**専用のラベル収集を別途作らず、これに相乗りするのが最善。**

| ベンチに必要 | リモート採点が持つ（`Models.swift` / `attempts`・`reviews`） |
|---|---|
| 画像 | `drawingData`(PencilKit) / Storage `drawing_path`（`.pkdrawing`）。※手書きは端末にローカル永続化済み（破棄されない） |
| target | `expected_word`（端末マスタ由来・自動） |
| **ラベル(ground_truth相当)** | **`parentReviewDecision` = approved / needsPractice（親の正誤判定）** |
| ローカルOCRベースライン | `recognized_text` |

- **親がレビューする＝ラベル付け**そのもの。手入力ゼロ。AI採点の目的が「親採点の代替」なので、**親判定を正解(ground truth)としてAIを評価**するのは方法論的にも正しい。
- 制約: `parentReviewDecision` は「正解/不正解」で**正確な綴り文字列ではない**。→ 最重要の**誤受理/誤拒否は正誤判定だけで測れる**。文字列レベルのOCR精度のみ別途（必要なら少数を文字起こし）。
- bench.py の横並びエンジンはそのまま流用し、**データ源を `attempts`/`reviews` に差し替える**。先に作った独立の `ocr_bench_samples` テーブルは不要（暫定/参考に格下げ）。

**依存（BUILT vs PLANNED — remote-grading-spec C/I より）**: attempts/reviews/手書きの同期は**未実装**（現状 words/profiles のみ同期）。2経路:
- **A. 暫定（バックエンド不要・最速）**: テスト端末で、ローカルの `practiceSamples`/`attempts` を PNG に描き出し＋親判定をラベルにしたファイルを書き出すデバッグ機能 → bench.py にローカル投入。Supabase不要・実使用＋実親ラベルで即計測。
- **B. 本筋**: 同期DTO(`AttemptDTO`/`ReviewDTO`)＋手書きStorage(`storage-sign`)が実装されたら、ベンチは Supabase の `attempts`/`reviews` を直接読む（製品としてどのみち作る部分）。

#### （参考）独立収集の設計（最良案が無理なときのフォールバック）

前提（コード確認済み 2026-06）:
- 採点時に `UIImage`＋出題語(`expected = currentWord.text`)＋ローカルOCR結果が揃う（`SpellingSessionView.swift` / `TestSessionView.swift`）。
- **`target`（出題語）は端末マスタと紐付き済み → 収集時に自動で分かる。手入力不要。**
- **現状、手書き画像はSupabaseに送っていない**（OCR用に一時生成して破棄）。画像アップロードは新規実装（収集フック）。

ラベル区別:
- `target` = 出題語（自動）
- `ground_truth` = 実際に書かれた文字（正解/誤字）。**誤受理を測るのに必要なのはこれ**。

**手入力をほぼゼロにする2系統の収集:**
1. **正解ケースは自動ラベル**: 普通に使ってもらい `ground_truth = target` を自動記録（"正しく書けた前提"フラグ＋ローカルOCR結果も保存）。→ OCR精度・**誤拒否**を大量にタダで測る。ランダムに数枚だけ目視監査し取りこぼし率を推定。
2. **誤字ケースは"指定収集モード"で自動ラベル**: アプリが「わざと『becuase』と書いてね」と提示。アプリは出題語(because)も書かせた文字(becuase)も知っているので **ground_truth を自動記録**。手入力ゼロで**誤受理**サンプルを量産。完全な自然使用ではないが手書き自体は本物。

⚠️ ①を `ground_truth=target` で自動ラベルすると「実は誤字なのに正解扱い」が紛れ込み**誤受理を取りこぼす**。だから誤受理は②（指定収集）で正面から測る、と役割分担する。

収集の置き場所（ツール側は実装済み）:
- `scripts/ocr-bench/schema.sql`: `ocr_bench_samples`（収集ファースト。`ground_truth`/`legible` は NULL 可＝未ラベル）＋ private バケット `ocr-bench`。アプリ同期スキーマとは分離。
- `bench.py` は `ground_truth IS NOT NULL`（ラベル済み）行だけを全モデルに投入。
- アプリ側の収集フック（`SupabaseService` にアップロード関数＋2採点画面でフック）は**設計のみ・未実装**（オプトインで自分＋同意テスターの端末だけ）。
