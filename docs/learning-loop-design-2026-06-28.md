# 学習ループ設計：つづり練習 × 文法クイズ × SRS（接点メモ）

Status: 検討中（戦略合意のための叩き台）。2026-06-28。
関連: おぼえかたヒント（`scripts/word_chunks.csv` ほか・本ブランチ）／
文法クイズ（別ワークツリー `feat/grammar-level-20260628`・`SentenceExercise.swift`/`GrammarLevel.swift`・**他エージェント作業中＝触らない**）。

## 0. なぜこのメモ
単語を「書くだけ（写経）」では定着が弱い。定着には4つが要る：
**①思い出す(テスト) ②構造化(かたまり) ③意味・運用(文で使う) ④間隔復習(SRS)**。
これらは別々の機能に散っている。**1つの学習ループ**として繋ぐための接点を先に決める。

### 統合の中心原則（いちばん大事・これだけ覚える）
> **単語に付く助け（おぼえかた＝かたまり ＋ 意味 ＋ 例文）は、その単語が出てくる所ならどこでも出る。**
> ＝ つづり練習でも、文法クイズでも、**同じ単語ヘルパー**が顔を出す（単語キーで繋ぐ＝codexの継ぎ目）。
- これにより「例文は文法クイズの先触れか？」という悩みは消える。例文をクイズと一致させる必要はない。
  単に**ヘルパーがどこでも一貫して出る**だけ。
- 出し方は「軽いものは出す／重いものは on-demand」：
  - つづり練習：例文(軽い1行)は画面に。かたまり(重い)はカード閲覧/誤答時。
  - 文法クイズ：**常時は出さない**（答えがバレる/ごちゃつく）。**タイルをタップした時・間違えた時**だけ単語ヘルパーを出す。

## 1. 単語のライフサイクル（1語が辿る道）
```
[出会い] 新出単語の導入
   ├ つづり練習（書く＝運動記憶）
   │    ├ おぼえかたヒント：かたまりで構造化（②）        ← カード閲覧/間違い時
   │    └ 例文チラ見：使われ方（③の予告）  ※新出語のみ・1短文 ← 導入と完了だけ
   ├ テスト（思い出す＝retrieval）（①）                   ← お手本なしで書く
   ▼
[運用] 文法クイズに同じ単語が出る（③の本番）
   ├ 並べ替え / 穴埋め / 手書き穴埋め(clozeHandwriting)
   │    └ 手書き穴埋めの誤答時：おぼえかたヒントを再利用      ← ここが両機能の接点
   ▼
[定着] SRS が「つづり」と「文での運用」を両方スケジュール（④）
```

## 2. 「例文を写経中に出すか」の決定 【決定済み 2026-06-28】
前提の訂正：本アプリの練習画面は **単語のお手本＋例文＋書くキャンバスが同一1画面**
（`SpellingSessionView.swift` の `wordHeader`→`ExampleHintView`＋下にキャンバス）。
「導入してから書く」という別フェーズは無い。＝**「最中」＝練習画面そのもの**。
よって「導入では出すが最中は出さない」という線引きは本アプリには成立しない。

決定：
- **例文は練習画面に出す**（出せる場所はそこだけ）。**日英1文ずつ・少し小さめ・対象語ハイライト**（Tanaka例文）。
- 軽い1行テキストなので書く邪魔にならない、というのが採用理由（情報過多にしない）。
- 出す対象：**新出語**中心（復習語は控える）。表示回（既存「最後のラウンドのみ／全ラウンド／出さない」設定）は実装時に調整。
- 区別の軸は「タイミング」ではなく**視覚的な重さ**：
  - **軽い1行（例文・意味）→ 練習画面でOK**。
  - **重いUI（おぼえかたヒントの色チップ）→ 練習画面には出さず**、①カード閲覧・②誤答時に置く。
  - （既存 `ExampleHintView` は en=19pt/ja=15pt。少し小さめへ調整余地）

## 2.5 レイアウトの大前提（後で詰める・最重要）
**デザインが本当に大事。** 練習画面は次を死守する：
- **単語が主役で大きい**（現状 `practiceWordHeader` は 48pt bold rounded・お手本が主役）。
- **例文は絶対に邪魔をしない**（小さい・控えめ・主役の単語を食わない）。視覚的階層を厳守。
- 具体レイアウト（例文の位置・サイズ・折りたたみ有無）は**別途デザイン回**で、実機スクショで決める。

## 3. 共有する“部品”（合体ではなく共有）
両機能を1つに混ぜない。代わりに **`SpellingSyncCore` の純粋部品を共有**：
| 部品 | つづり練習 | 文法クイズ | 備考 |
|---|---|---|---|
| **学年の壁ゲート** | 中学生ゲート | `GrammarGate`/`gradeCeiling` | 概念が同じ。基準を揃える（CEFR-J×学年）|
| **おぼえかたヒント**（かたまり）| カード/誤答時 | clozeHandwriting 誤答時 | `MorphemeHint.assemble` を両者が呼ぶ |
| **語彙の壁**（NGSL/band）| 既知語プール | `SentenceSelection.gradeBand` | 既に文法側にある。共通化 |
| **SRS** | つづりの定着 | 文での運用 | 同じ単語IDで両イベントを記録 |

## 3.5 統合アーキテクチャ方針 【codex Architect レビュー反映・2026-06-28】
※ 文法クイズ(sentence-builder)は **既に main にマージ済み**（`SentenceExercise.swift`/`GrammarLevel.swift`）。

**結論：合体（モノリス）ではなく「背骨を共有・機能はモジュール」。** 概念は1つの学習システム、実体は薄い継ぎ目。
1. **学年ゲートは `GrammarStage` を流用**（新しい学年型を作らない）。ただし名前は `minimumLearnerStage`（例 `.basic1`=中1）にして、
   「**語の難しさ**＝単語データ(承認行/level/長さ/family安全性)」と「**学年の壁**＝GrammarStage」を**混同しない**。GrammarStage は「この子にこの見せ方を出してよい年齢か」を答えるだけ。
2. **継ぎ目＝単語キーで繋ぐ小さなコーディネータ**（新ファイル `WordUsageLinker.swift`）。
   `eligibleSentences(wordKey, targetBand, grammarCeiling)`：練習した単語を含む `SentenceItem`（`contentLemmas`）を、既存 `SentenceSelection`＋`GrammarGate` で絞って返す。
   **共有 `LeveledWord` モデルは今は作らない**（早すぎる）。`SentenceItem` が既に必要なフックを持つ。
3. **データは別ソース＋共通契約**：かたまりは wordbank.sqlite、文は sentence bank のまま。共有するのは退屈な土台だけ
   （**正規化した単語キー**・wordbank存在チェック・NGSL/band参照・検証レポート）。ビルド時に「chunk.word と sentenceのcontentLemmas が同じ正規化キー」を検査。
4. **SRSは1エンジン・状態は別**：複合キー `(word, skill: spelling|usage)`。つづり成功で運用が勝手に進まない／文のミスでつづりを罰しない。
5. **収束はビルド時の継ぎ目追加で**（big-bang マージしない）。**既存の文法ファイルは触らない**（別エージェント作業中）。橋渡しは新ファイルのみ。

**主なリスクと対策**：軸の混同→`minimumLearnerStage`命名／**join ずれ**（inspect vs inspects・大文字小文字）→**正規化キー1本化＋ビルド時検証**／文法側との競合→新ファイルのみ／SRS汚染→(word,skill)で分離／データ品質→承認CSV＋allowlist＋連結検証＋nil非表示。

## 4. 接点インターフェイス（今は実装しない・合流可能にだけしておく）
- おぼえかたヒントは **単語文字列を入力に取り、表示モデルを返す純関数**として作る
  （`MorphemeHint.assemble(word, rows) -> [MorphemePart]?`）。UIに依存しない＝文法クイズからも呼べる。
- 単語の「出会い→運用」を繋ぐため、**同じ語彙ID**で参照する（wordbank の word をキーにする）。
- SRS は将来 `(wordID, skill: .spelling | .usage)` の2軸で記録できる余地を残す（今は spelling のみでよい）。
