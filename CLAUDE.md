# SpellingTrainer — プロジェクトガイド

日本人の子ども向け 英単語スペル練習 iPad アプリ（SwiftUI / iOS 17 / Swift 6）。

## 作業フロー（Git・ワークツリー運用）（重要）

複数のエージェントが同じリポジトリで**同時に動いている前提**で作業する。

- **専用ブランチ＋ワークツリーで作業する**。`main`（デフォルトブランチ）では直接作業・コミットしない。
  - 例: `git worktree add ../SpellingTrainer-wt/<branch> -b <branch>`
  - ワークツリーはリポジトリ外の専用ディレクトリに置く（リポジトリ内にネストしない）。
- **必ずブランチを切ってから作業する**。`main` に直接コミットしない。作業前に新しいブランチを作成すること。
- **名前は衝突しないよう最大限配慮する**。他エージェントと被らないこと。
  - **ブランチ名の競合を避ける**：内容を表すプレフィクス＋短いユニークな suffix（例: `feat/home-characters-<日付 or 短いランダム>`）。作成前に `git branch --list` で既存ブランチと被らないか必ず確認する。
  - **マイグレーションファイルの競合を避ける**：スキーマ/データ移行ファイル名（マイグレーション名）も同様にユニークな suffix を付け、作成前に既存の同種ファイルと被らないか必ず確認する。複数エージェントが同時にマイグレーションを足すと番号・名前が衝突するため特に注意。
  - ワークツリーのディレクトリ名も同じ方針で被らせない。
- **クローズ時（マージ/PR 反映後）は後片付けする**。残骸を残さない。
  - `git worktree remove <path>` でワークツリー削除 → `git branch -d <branch>` でブランチ削除。
- 例外: ごく軽微な単発確認などで、ユーザーが明示的に `main` 上での作業を許可した場合のみ直接作業可。

## 実装ルール（重要）

- **TDD で開発する**。新しいロジック・バグ修正・リファクタは、原則 **テストを先に書いて失敗(RED)→最小実装(GREEN)→リファクタ** の順で進める。
  - **テスト可能なロジックは SwiftPM パッケージ `SpellingSyncCore` に置く**（`swift test` で高速に回る／カバレッジ目標 80%+、重要ロジックは 100%）。同期の純粋ロジック（LWW・tombstone・SRS・移行・カーソル/差分選択など）はここ。
  - アプリ本体（`iPadPrototype`、Supabase I/O・UI）は **薄く保つ**。現状 XCTest ターゲットが無いため、I/O は手動/結合テストで確認する（アプリ単体テストが必要になったら XCTest ターゲットを Xcode で追加する）。
  - 純粋ロジックと I/O を分離し、「テストが書けない形」で本質ロジックを書かない。
- **実装後は必ずレビューさせる**。コード変更を終えたら、コミット/PR の前に必ずレビューを通すこと（codex の Code Reviewer などに依頼）。レビュー指摘は対応してから次に進む。
- **指示されていないファイルを修正しない**。タスクで指定された範囲のファイルだけを変更する。関連して触りたくなったファイルがあっても、勝手に変更せずユーザーに確認する。

## UI/UX 設計思想（重要）

### 基本の軸：2人のユーザーを分ける
このアプリには性質の違う2人のユーザーがいる。常にこの軸で考える。

| | 子ども | 親 |
|---|---|---|
| 役割 | **やる人**（doer） | **管理する人**（manager） |
| 見せるもの | 今やること＋ごほうび/反応 | 単語・レベル・採点・記録・設定 |
| トーン | 大きい・少ない・遊び心・ふりがな | 情報密度OK・数字/バッジOK |
| 専門用語 | ❌ 出さない（Lv / CEFR / Dolch / 採点 等） | ⭕️ OK |

**迷ったら問う:** 「これは“やる人”が見る/操作する必要がある？ それとも“管理”？」
どちらもNoなら**親側**に置く。

### 子ども側の原則
1. **1画面 = 1メイン動作**（例: はじめる）。選択肢を増やさない
2. **大きいタップ＋即フィードバック**（発音・バウンス・アイリス遷移・キャラ）
3. **文字で説明しない**。アイコン＋ふりがな＋音で伝える
4. **複雑さは親ゲートの奥に隠す**（レベル・採点・設定）
5. **段階的開示**：普段は隠し、触った時だけ出す（例: たんごカードの例文アコーディオン）
6. ラベリングしない。レベルや評価を子に見せて優劣を意識させない

### 親側の原則
- ここは**管理コンソール**。情報密度・数字・バッジは許容
- **タスクで分類**：単語管理／採点／記録／設定
- レベルは「単語管理の中の“作る手段の一つ”」として置く（例: 「レベルで作成」）
- レベルバッジを出すなら**親の一覧だけ**。子の画面には出さない

### 新機能を足すときの問い
- 子に**見せる**？ 子に**操作させる**？ → どちらもNoなら親側へ
- 既存の見た目言語（色分け・角丸カード・バッジ・バウンス）を**再利用**できるか？
  新しい見た目を不用意に増やさない

### レベル機能の落とし所（決定事項）
- レベル（Dolch=US学年 / NGSL=難易度）は **親側の「セット生成ツール」専用**
- **子のたんご画面・ホームには出さない**
- 将来やるとしても親の「全単語」一覧に控えめなバッジ程度まで

## 同梱データ
`iPadPrototype/Resources/wordbank.sqlite`（読み取り専用、`WordBank`からアクセス）
- `gloss`: 英→和の訳語（EJDict / パブリックドメイン）
- `examples`: 例文 英＋日（Tanaka Corpus / CC BY 2.0 FR）。**子ども不適切な内容は除外済み**
- `level`: 単語→Dolch学年＋NGSL難易度バンド
- クレジット表記は保護者メニュー→設定に掲載（Tanaka CC-BY / EJDict PD）

## 開発メモ
- ビルド確認は `xcodebuild ... -sdk iphonesimulator build CODE_SIGNING_ALLOWED=NO`
- `wordbank.sqlite` は静的リソース（差分が無ければ再転送されない）
- `.claude/` は git 管理外（gitignore 済み）

## 画像生成（codex）
- **codex（`mcp__codex__codex`）は built-in の `image_gen` ツールで実際にラスター画像（PNG）を生成できる**。背景・イラスト等のアセットはこれで作れる（外部の画像生成APIキーは不要）。
- 依頼方法: `sandbox: workspace-write` ＋ 出力先を `cwd` に指定し、プロンプトで「`image_gen` で生成し PNG ファイルとして保存」「縦横比・スタイル・ファイル名」を明示する。生成後に `ls -la` / `file` で確認させると確実。
- 実績: 子ども向けホーム背景 1448×1086 PNG を 10 種生成（やわらかいフラットイラスト・中央を開けてキャラを乗せる構図）。背景の追加候補は管理用 CSV（`scripts/backgrounds.csv` 想定）の `art_prompt` 列を生成プロンプトに使う。
