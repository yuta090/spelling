# SpellingTrainer — プロジェクトガイド

日本人の子ども向け 英単語スペル練習 iPad アプリ（SwiftUI / iOS 17 / Swift 6）。

## 公開状況（重要）

- **本アプリはまだ一般公開していない（App Store 未リリース・実ユーザーのデータが存在しない）**。
- したがって **既存のコンテンツ・データ・スキーマ・ID を自由に変更してよい**。公開ユーザー向けの慎重な後方互換・段階的データ移行・本番データ保護は **現時点では不要**（破壊的変更・作り直し・ファイル再生成を恐れない）。
  - 例：同梱 JSON/DB のフォーマット変更、表層 ID の振り直し、`sentence_bank.json` 等の作り直しは自由にやってよい（決定論ビルドで差分が安定することだけ確認する）。
  - ただし **コードの設計品質（テスト・レビュー・純ロジック分離）は通常どおり維持**する。「自由に変えてよい」のは“移行の慎重さ”の話であって、品質を落としてよいという意味ではない。
- 公開（リリース）したらこの前提は変わる。**リリース後は実ユーザーのデータ移行・後方互換を必ず考慮する**。状況が変わったらこの節を更新する。

## 作業フロー（Git・ワークツリー運用）（重要）

複数のエージェントが同じリポジトリで**同時に動いている前提**で作業する。

- **`main` での作業は禁止（例外なし）**。`main` ブランチ上で編集・コミットしない。軽微な確認・1行修正でも必ずブランチを切る。`main` は PR マージと同期（pull）専用。
- **専用ブランチ＋ワークツリーで作業する**。`main`（デフォルトブランチ）では直接作業・コミットしない。
  - 例: `git worktree add ../SpellingTrainer-wt/<branch> -b <branch>`
  - ワークツリーはリポジトリ外の専用ディレクトリに置く（リポジトリ内にネストしない）。
- **🚫 本体（メイン）リポジトリフォルダでの `git checkout` / `git switch` は禁止（例外なし）**。作業ブランチへ移るのに**本体フォルダのブランチを切り替えてはいけない**。必ず `git worktree add` で**別フォルダ**を作り、そこで作業する。
  - 理由: 本体フォルダはユーザーが Xcode で開きっぱなしにする共有スペース。ここでブランチを切り替えると、**他のエージェントやユーザーが見ている画面のブランチが勝手に変わり**、ビルド内容が入れ替わって混乱する（実害として「さっき入れた変更が消えた／出ない」が起きる）。
  - **本体フォルダは常に `main` に置いたままにする**（用途は PR マージと `git pull` 同期のみ）。作業で `git checkout <branch>` したくなったら、代わりに `git worktree add ../SpellingTrainer-wt/<branch> -b <branch>` する。
  - 同様に、本体フォルダで feature ブランチを **作成（`-b`）・コミット・マージのチェックアウトをしない**。マージは `main` 上で行い、作業は worktree 側で行う。
  - 自分が今どこにいるか不安なら、作業前に `git rev-parse --show-toplevel` と `git branch --show-current` で「専用 worktree の・専用ブランチ上」にいることを確認してから編集する。
- **必ずブランチを切ってから作業する**。`main` に直接コミットしない。作業前に新しいブランチを作成すること。
- **名前は衝突しないよう最大限配慮する**。他エージェントと被らないこと。
  - **ブランチ名の競合を避ける**：内容を表すプレフィクス＋短いユニークな suffix（例: `feat/home-characters-<日付 or 短いランダム>`）。作成前に `git branch --list` で既存ブランチと被らないか必ず確認する。
  - **マイグレーションファイルの競合を避ける**：スキーマ/データ移行ファイル名（マイグレーション名）も同様にユニークな suffix を付け、作成前に既存の同種ファイルと被らないか必ず確認する。複数エージェントが同時にマイグレーションを足すと番号・名前が衝突するため特に注意。
  - ワークツリーのディレクトリ名も同じ方針で被らせない。
- **クローズ時（マージ/PR 反映後）は後片付けする**。残骸を残さない。
  - `git worktree remove <path>` でワークツリー削除 → `git branch -d <branch>` でブランチ削除。
- **作業完了後は、次の一連を実施してよいか必ずユーザーに確認する**（勝手に進めない・勝手にコミットしない）。
  - ①コミット → ②PR 作成 → ③マージ → ④`main` 更新（pull/同期）→ ⑤ワークツリーとブランチの後片付け。
  - まとめて「コミット〜マージ〜後片付けまで進めてよいですか？」と1行で確認し、承認を得てから実行する。
  - マージ時に `main` が進んで競合した場合は取り込んで解消し、ビルド／テストを通してから進める。

## 実装ルール（重要）

- **TDD で開発する**。新しいロジック・バグ修正・リファクタは、原則 **テストを先に書いて失敗(RED)→最小実装(GREEN)→リファクタ** の順で進める。
  - **テスト可能なロジックは SwiftPM パッケージ `SpellingSyncCore` に置く**（`swift test` で高速に回る／カバレッジ目標 80%+、重要ロジックは 100%）。同期の純粋ロジック（LWW・tombstone・SRS・移行・カーソル/差分選択など）はここ。
  - アプリ本体（`iPadPrototype`、Supabase I/O・UI）は **薄く保つ**。現状 XCTest ターゲットが無いため、I/O は手動/結合テストで確認する（アプリ単体テストが必要になったら XCTest ターゲットを Xcode で追加する）。
  - 純粋ロジックと I/O を分離し、「テストが書けない形」で本質ロジックを書かない。
- **書いたコードは必ず codex にレビューさせる（ハードゲート・例外なし）**。コード変更を終えたら、**コミット/PR の前に必ず `mcp__codex__codex`（Code Reviewer）で独立レビューを通す**。**レビューを通す前に「完了」と報告しない**。レビュー指摘は対応してから次に進む。サブエージェントに実装させた場合も、そのエージェントの完了ゲートに codex レビューを含める（各 `.claude/agents/*.md` に明記済み）。
- **指示されていないファイルを修正しない**。タスクで指定された範囲のファイルだけを変更する。関連して触りたくなったファイルがあっても、勝手に変更せずユーザーに確認する。

## モデル役割分担（重要）

複数モデル（Fable5 / Opus4.8 / Sonnet5 / Haiku4.5）を**業務の型で使い分ける**。狙いは Fable5 の消費を最適化すること——**過剰消費も過剰節約もしない**。思想は「**Fable を動かす前に、下位モデルで準備（収集・整形・検査）を全部済ませ、机が完璧になった状態で“判断だけ”を Fable に渡す**」。

### モデルの担当
- **Fable5** … 「**型がない × 失敗コストが大きい × 全体を見る必要がある**」判断だけ。新機能の設計・仕様策定・アーキテクチャの分岐・非定型な大きい競合解消。**メインセッション専用**。
- **Opus4.8** … 日常の判断・レビュー・**執筆**・コア実装（TDD）・自然さ判定・PR/docs 執筆。**メインの既定**。
- **Sonnet5** … 型が決まった**量産・実行**（コンテンツ追記、設計確定後のアプリ配線/I/O、テスト量産）。
- **Haiku4.5** … **機械的な変換・検査・収集・実行**（ビルド/テスト、決定論diff、band/漢字/重複CLI、在庫点検、探索、Git定型手続き、アセットの切抜/検証）。

### 動かし方（振り分けは基本“自動”）
- **サブエージェント**＝ `.claude/agents/*.md` の `model:` で自動切替。既定エージェント：
  - `content-author`(sonnet)＝コンテンツ量産 / `content-naturalness-reviewer`(opus)＝自然さ監査 / `asset-runner`(haiku)＝画像・Lottie実処理 / `inspector`(haiku)＝検査・在庫・ビルド/テスト・探索。
- **Workflow** ＝ 台本内で各 `agent()` に `model`/`effort` を指定して自動振り分け。
- **スキル** ＝ 手順書にモデルは書かず、上のエージェントを呼ぶ形に一元化（各 SKILL.md の「実行モデル（振り分け）」参照）。

### メインセッションのモデル切替（＝私が /model で手動、Claudeは提案まで）
- メインのモデルは**仕様上 Claude 自身では変えられない**。Claude は「**今 Fable に切り替えるべきです／Opus に戻して良いです**」と**提案するだけ**。ユーザーが `/model` で切り替える。
- 提案ルール：メインは **Opus 常駐**。**非定型な設計判断の局面に入る直前**に「Fable に切替を」と提案し、その判断が済んだら「Opus に戻して良いです」と提案する（昇格は最小限・都度戻す）。
- **Fable を呼ぶ前に机を整える**：Fable に設計を渡す前に、`inspector`(haiku) で在庫点検・関連コード収集・ビルド/テスト緑を確認し、Sonnet/Opus で叩き台・選択肢整理まで用意して「**判断だけ**」を渡す。

### コードレビュー（再掲・ハードゲート）
- **書いたコードは必ず codex（`mcp__codex__codex`）レビューを通す**。実装エージェントの完了ゲートにも含める（上「実装ルール」参照）。

> 単発・未確定の業務は定義ファイル化しない（この思想ルールに留める）。**1回やって繰り返すと分かったら**、`.claude/agents/` かスキル手順書に固定する。

## UI/UX 設計思想（重要）

### 基本の軸：2人のユーザーを分ける
このアプリには性質の違う2人のユーザーがいる。常にこの軸で考える。

| | 子ども | 親 |
|---|---|---|
| 役割 | **やる人**（doer） | **管理する人**（manager） |
| 見せるもの | 今やること＋ごほうび/反応 | 単語・レベル・採点・記録・設定 |
| トーン | 大きい・少ない・遊び心・ふりがな | 情報密度OK・数字/バッジOK |
| 専門用語 | ❌ 出さない（Lv / CEFR / Dolch / 採点 等） | ⭕️ OK |

**迷ったら問う:** 「これは“やる人”が見る/操作する必要がある？ それとも“管理”？」
どちらもNoなら**親側**に置く。

### 子ども側の原則
1. **1画面 = 1メイン動作**（例: はじめる）。選択肢を増やさない
2. **大きいタップ＋即フィードバック**（発音・バウンス・アイリス遷移・キャラ）
3. **文字で説明しない**。アイコン＋ふりがな＋音で伝える
4. **複雑さは親ゲートの奥に隠す**（レベル・採点・設定）
5. **段階的開示**：普段は隠し、触った時だけ出す（例: たんごカードの例文アコーディオン）
6. ラベリングしない。レベルや評価を子に見せて優劣を意識させない

### 親側の原則
- ここは**管理コンソール**。情報密度・数字・バッジは許容
- **タスクで分類**：単語管理／採点／記録／設定
- レベルは「単語管理の中の“作る手段の一つ”」として置く（例: 「レベルで作成」）
- レベルバッジを出すなら**親の一覧だけ**。子の画面には出さない

### 新機能を足すときの問い
- 子に**見せる**？ 子に**操作させる**？ → どちらもNoなら親側へ
- 既存の見た目言語（色分け・角丸カード・バッジ・バウンス）を**再利用**できるか？
  新しい見た目を不用意に増やさない

### レベル機能の落とし所（決定事項）
- レベル（Dolch=US学年 / NGSL=難易度）は **親側の「セット生成ツール」専用**
- **子のたんご画面・ホームには出さない**
- 将来やるとしても親の「全単語」一覧に控えめなバッジ程度まで

## コンテキスト管理（`/compact` を勧める）（重要）
- 会話が長くなり**コンテキスト圧縮が有効そうなタイミングでは、ユーザーに `/compact` の実行を推奨する**（こちらが勝手に実行はしない。推奨に留める）。
- 勧めるべき主なタイミング:
  - **タスクの区切り**（PR 作成/マージ完了・ワークツリー後片付け済みなど、作業が一段落した時）。
  - **新しい大きめタスクに着手する直前**（前タスクの詳細な探索ログを引きずらず、クリーンな状態で始めたい時）。
  - 長時間/多数のツール出力（大きな diff・ビルドログ・探索結果）が積み上がり、文脈が冗長になってきたと感じた時。
- 勧め方: 「ここで `/compact` すると軽くなります」と**1行で短く**提案する。直前に未コミットの変更や保存すべき判断が無いか（メモリ/HANDOFF への記録漏れ）を一言添える。しつこく繰り返さない（断られたら同一トピック中は再提案しない）。

## 作業の区切り報告（重要）
- **タスクが一段落したら、聞かれなくても「何をやったか／次に何をすべきか」を簡潔に報告する**。
  - **やったこと**: 今回の作業を箇条書きで短く（成果・確認結果＝テスト/ビルド・push/マージ状況など）。
  - **次にやるべきこと**: 残タスクがあれば短く列挙。**本当に何も無ければ「特にありません」とはっきり言う**（無理に作らない・タスクをでっち上げない）。
- 冗長な経緯の再説明はしない。長くせず要点だけ。続けるなら `/compact` 推奨も一言添えてよい（→「コンテキスト管理」参照）。

## 画像生成（codex）
- **codex（`mcp__codex__codex`）は built-in の `image_gen` ツールで実際にラスター画像（PNG）を生成できる**。背景・イラスト等のアセットはこれで作れる（外部の画像生成APIキーは不要）。
- 依頼方法: `sandbox: workspace-write` ＋ 出力先を `cwd` に指定し、プロンプトで「`image_gen` で生成し PNG ファイルとして保存」「縦横比・スタイル・ファイル名」を明示する。生成後に `ls -la` / `file` で確認させると確実。
- 実績: 子ども向けホーム背景 1448×1086 PNG を 10 種生成（やわらかいフラットイラスト・中央を開けてキャラを乗せる構図）。背景の追加候補は管理用 CSV（`scripts/backgrounds.csv` 想定）の `art_prompt` 列を生成プロンプトに使う。
